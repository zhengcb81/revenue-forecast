# 多维度防跳过机制详解

**版本**: v2.5.0  
**创建日期**: 2026-03-01

---

## 问题背景

### 传统时间检测的问题

在 v2.5.0 之前的版本中，我们使用了**硬性时间限制**来防止偷懒：

```python
# 旧方法：硬性时间限制
if execution_time < 60:  # 必须执行超过60秒
    raise Error("执行时间过短，可能存在偷懒")
```

**这个方法的问题**:

1. **模型速度提升**
   - 随着模型优化，同样工作可能耗时更少
   - 一个高效的模型可能在30秒内完成原本需要60秒的工作
   - 结果：误报，阻止了正常的快速执行

2. **缓存命中**
   - 如果部分数据已缓存，查询时间会显著缩短
   - 重复分析同一家公司时，速度会更快
   - 结果：误报，即使工作已经完成

3. **任务复杂度差异**
   - 简单公司 vs 复杂公司的分析时间差异巨大
   - 硬性时间无法适应不同场景

---

## 新方案：多维度综合验证

### 核心理念

**不关注"花了多长时间"，而关注"实际做了什么"**:

```
工作量 = Token消耗 + 内容产出 + 工具调用 + 文件生成
```

### 四个验证维度

| 维度 | 衡量指标 | 阈值示例 | 优势 |
|-----|---------|---------|------|
| **Token** | 思考深度 | step4: 8000+ | 不受速度影响 |
| **内容** | 输出质量 | step4: 5000+字符 | 直接反映质量 |
| **工具** | 行动次数 | step4: 18+ | 最可靠指标 |
| **文件** | 工作成果 | step4: 9个文件 | 客观可复查 |

---

## 综合通过逻辑

### 关键项必须 + 其他互补

```
硬性要求（必须满足）：
✅ 内容深度达标（防止生成垃圾）
✅ 文件生成完整（确保交付物）

软性要求（可互补）：
⚪ Token消耗达标
⚪ 工具调用达标

通过条件：
硬性要求全部满足 + 软性要求至少满足一项
```

## 示例场景

### 场景1：速度快但产出充足（通过）

```
模型速度很快，30秒完成step4：
- Token: 12000 ✅
- 内容: 6000字符 ✅
- 工具调用: 22次 ✅
- 文件: 9个维度文件 ✅

结果：通过（虽然时间很短，但产出充足）
```

### 场景2：想偷懒被检测（失败）

```
试图跳过搜索直接生成内容：
- Token: 2500 ❌
- 内容: 1500字符 ❌
- 工具调用: 2次 ❌
- 文件: 2个维度文件 ❌

结果：失败（明显偷懒）
```

## 总结

### 新旧方案对比

| 维度 | 旧方案 | 新方案 |
|-----|-------|-------|
| 核心指标 | 执行时间 | Token+内容+工具+文件 |
| 速度敏感 | 是 | 否 |
| 缓存敏感 | 是 | 否 |
| 误报率 | 高 | 低 |

### 核心优势

1. **公平性**: 速度快的模型也能通过，只要产出达标
2. **可靠性**: 多重检查，难以绕过
3. **适应性**: 适应不同复杂度场景
4. **可复查**: 文件和记录可以事后验证
